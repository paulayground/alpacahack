# 배운 점

- `>>`를 통해 비트가 버려지며 원본값에 어떤 비트들이 있었는 지를 알 수 없기 때문에 `<<`를 통하여 원본값을 정확하게 도출해낼 수 없다.
- 컴파일러는 나눗셈/나머지 연산이 느리기 때문에 곱셉과 시프트연산으로 변환하여 더 빨리 처리한다. 이 때 `(x * M) >> s(M=매직넘버, s=shift값)`과 같은 연산이 사용된다.
  ```c
  // 5의 매직넘버 3435973837
  // y = x / 5
  uint64_t y = (x * 3435973837) >> 34;
  // 7의 매직넘버 613566757
  // t = x / 7
  uint64_t t = (x * 613566757) >> 32;
  // z = x % 7 과 같다
  uint64_t z = x - ((((x - t) >> 1) + t) >> 2) * 7;
  ```

# 풀이 과정 기록

brute force보다는 계산로직을 역으로 풀려고 했는데, 실행결과 서로 계산의 값이 달라 확인한 결과 `>>`를 통해 비트가 버려지기 때문에 `<<`를 통해서도 항상 정확한 값을 가져올 수 없었다. 그렇기 때문에 brute force를 통한 방식을 사용하였다.
